159 // This might be the most ugliest hack I've done to fix a Windows
160 // bug. Here's the thing:
161 // 1) When we use the pen on a Surface device, it send us
162 // WM_POINTERUPDATE messages, but after a WM_POINTERUPDATE
163 // we'll receive a WM_MOUSEMOVE with the position of the mouse
164 // (which is different to the position of the pen). This
165 // happens constantly if we press a modifier key like Alt,
166 // Ctrl, or Shift, and randomly if we don't use a modifier.
167 // 2) First I tried to fix this issue disabling the WM_MOUSEMOVE
168 // processing when a pointer device is inside the window (with
169 // WM_POINTERENTER/LEAVE events), but that generated some buggy
170 // behavior (sometimes we stopped processing WM_MOUSEMOVE
171 // messages completely, it looks like a WM_POINTERLEAVE could
172 // be lost in time).
173 // 3) Then I tried to fix this using the Windows pointer API for
174 // mouse messages (EnableMouseInPointer(TRUE)) but that API
175 // doesn't work well with Wacom drivers. And I guess it might
176 // depend on the specific driver version.
177 // 4) Now I've reverted the pointer API change (we keep using the
178 // WM_MOUSE* messages) and fixed the original issue with this
179 // "m_ignoreRandomMouseEvents" variable. This is a counter that
180 // indicates the number of WM_MOUSEMOVE messages that we must
181 // ignore after a WM_POINTERUPDATE.
182 // 5) Each time we receive a WM_POINTERUPDATE this counter is set
183 // to 2. A value that I calculated practically just testing the
184 // pen in a Surface Pro using the Alt modifier and seeing how
185 // many random WM_MOUSEMOVE messages were received.
186 // 6) When a WM_MOUSEMOVE is received this counter is decreased,
187 // so when it backs to 0 we start processing WM_MOUSEMOVE
188 // messages again.
189 int m_ignoreRandomMouseEvents;
190
191 // Variables used to convert one finger in mouse-like movement,
192 // and two/more fingers in scroll movement/pan/zoom. The idea
193 // is as follows:
194 // 1) When a PT_TOUCH is received, we count this event in
195 // m_fingers and setup a timer (m_fingerTimerID) to wait for
196 // another touch event.
197 // 2) If another touch event is received, we process the messages
198 // with the interaction context (m_ictx) to handle special
199 // gestures (pan, magnify, etc.).
200 // 3) If the timeout is reached, and we've received only one
201 // finger on the windows, we can send all awaiting events
202 // (m_fingerEvents) like mouse movement messages/button
203 // presses/releases.
204 struct Touch {
205 int fingers; // Number of fingers in the window
206 // True when the timeout wasn't reached yet and the finger can be
207 // converted to mouse events yet.
208 bool canBeMouse;
209 // True if we're already processing finger/touch events as mouse
210 // movement events.
211 bool asMouse;
212 // Timeout (WM_TIMER) when the finger is converted to mouse events.
213 UINT_PTR timerID;
214 // Queued events to be sent when the finger is converted to mouse
215 // events (these events are discarded if another finger is
216 // introduced in the gesture e.g. to pan)
217 std::vector<Event> delayedEvents;
218 Touch();
219 } *m_touch;
