 \hypertarget{md__ase_sprite_aseprite__i_n_s_t_a_l_l_autotoc_md75}{}\doxysection{Compiling}\label{md__ase_sprite_aseprite__i_n_s_t_a_l_l_autotoc_md75}

\begin{DoxyEnumerate}
\item Get Aseprite code, put it in a folder like {\ttfamily C\+:\textbackslash{}aseprite}, and create a {\ttfamily build} directory inside to leave all the files that are result of the compilation process ({\ttfamily .exe}, {\ttfamily .lib}, {\ttfamily .obj}, {\ttfamily .a}, {\ttfamily .o}, etc). \begin{DoxyVerb} cd C:\aseprite
 mkdir build
\end{DoxyVerb}
 In this way, if you want to start with a fresh copy of Aseprite source code, you can remove the {\ttfamily build} directory and start again.
\item Enter in the new directory and execute {\ttfamily cmake}\+: \begin{DoxyVerb} cd C:\aseprite\build
 cmake -G Ninja -DLAF_BACKEND=skia ..
\end{DoxyVerb}
 Here {\ttfamily cmake} needs different options depending on your platform. You must check the details for Windows, mac\+OS, and Linux. Some {\ttfamily cmake} options can be modified using tools like \href{https://cmake.org/cmake/help/latest/manual/ccmake.1.html}{\texttt{ {\ttfamily ccmake}}} or \href{https://cmake.org/cmake/help/latest/manual/cmake-gui.1.html}{\texttt{ {\ttfamily cmake-\/gui}}}.
\item After you have executed and configured {\ttfamily cmake}, you have to compile the project\+: \begin{DoxyVerb} cd C:\aseprite\build
 ninja aseprite
\end{DoxyVerb}

\item When {\ttfamily ninja} finishes the compilation, you can find the executable inside {\ttfamily C\+:\textbackslash{}aseprite\textbackslash{}build\textbackslash{}bin\textbackslash{}aseprite.\+exe}.
\end{DoxyEnumerate}\hypertarget{md__ase_sprite_aseprite__i_n_s_t_a_l_l_autotoc_md76}{}\doxysubsection{Windows details}\label{md__ase_sprite_aseprite__i_n_s_t_a_l_l_autotoc_md76}
Open a \href{https://docs.microsoft.com/en-us/dotnet/framework/tools/developer-command-prompt-for-vs}{\texttt{ developer command prompt}} or in the command line ({\ttfamily cmd.\+exe}) call\+: \begin{DoxyVerb}call "C:\Program Files\Microsoft Visual Studio\2022\Community\Common7\Tools\VsDevCmd.bat" -arch=x64
\end{DoxyVerb}
 And then \begin{DoxyVerb}cd aseprite
mkdir build
cd build
cmake -DCMAKE_BUILD_TYPE=RelWithDebInfo -DLAF_BACKEND=skia -DSKIA_DIR=C:\deps\skia -DSKIA_LIBRARY_DIR=C:\deps\skia\out\Release-x64 -DSKIA_LIBRARY=C:\deps\skia\out\Release-x64\skia.lib -G Ninja ..
ninja aseprite
\end{DoxyVerb}
 In this case, {\ttfamily C\+:\textbackslash{}deps\textbackslash{}skia} is the directory where Skia was compiled or uncompressed.\hypertarget{md__ase_sprite_aseprite__i_n_s_t_a_l_l_autotoc_md77}{}\doxysubsubsection{Min\+GW}\label{md__ase_sprite_aseprite__i_n_s_t_a_l_l_autotoc_md77}
We don\textquotesingle{}t support Min\+GW compiler and it might bring some problems into the compilation process. If you see that the detected C++ compiler by cmake is {\ttfamily C\+:\textbackslash{}Min\+GW\textbackslash{}bin\textbackslash{}c++.exe} or something similar, you have to get rid of Min\+GW path ({\ttfamily C\+:\textbackslash{}Min\+GW\textbackslash{}bin}) from the {\ttfamily PATH} environment variable and run cmake again from scratch, so the Visual Studio C++ compiler ({\ttfamily cl.\+exe}) is used instead.

You can define the {\ttfamily CMAKE\+\_\+\+IGNORE\+\_\+\+PATH} variable when running cmake for the first time in case that you don\textquotesingle{}t know or don\textquotesingle{}t want to modify the {\ttfamily PATH} variable, e.\+g.\+: \begin{DoxyVerb}cmake -DCMAKE_IGNORE_PATH=C:\MinGW\bin ...
\end{DoxyVerb}
 More information in \href{https://github.com/aseprite/aseprite/issues/2449}{\texttt{ issue \#2449}}\hypertarget{md__ase_sprite_aseprite__i_n_s_t_a_l_l_autotoc_md78}{}\doxysubsection{mac\+OS details}\label{md__ase_sprite_aseprite__i_n_s_t_a_l_l_autotoc_md78}
Run {\ttfamily cmake} with the following parameters and then {\ttfamily ninja}\+: \begin{DoxyVerb}cd aseprite
mkdir build
cd build
cmake \
  -DCMAKE_BUILD_TYPE=RelWithDebInfo \
  -DCMAKE_OSX_ARCHITECTURES=x86_64 \
  -DCMAKE_OSX_DEPLOYMENT_TARGET=10.9 \
  -DCMAKE_OSX_SYSROOT=/Applications/Xcode.app/Contents/Developer/Platforms/MacOSX.platform/Developer/SDKs/MacOSX.sdk \
  -DLAF_BACKEND=skia \
  -DSKIA_DIR=$HOME/deps/skia \
  -DSKIA_LIBRARY_DIR=$HOME/deps/skia/out/Release-x64 \
  -DSKIA_LIBRARY=$HOME/deps/skia/out/Release-x64/libskia.a \
  -G Ninja \
  ..
ninja aseprite
\end{DoxyVerb}
 In this case, {\ttfamily \$HOME/deps/skia} is the directory where Skia was compiled or downloaded. Make sure that {\ttfamily CMAKE\+\_\+\+OSX\+\_\+\+SYSROOT} is pointing to the correct SDK directory (in this case {\ttfamily /\+Applications/\+Xcode.app/\+Contents/\+Developer/\+Platforms/\+Mac\+OSX.\+platform/\+Developer/\+SDKs/\+Mac\+OSX.sdk}), but it could be different in your Mac.\hypertarget{md__ase_sprite_aseprite__i_n_s_t_a_l_l_autotoc_md79}{}\doxysubsubsection{Apple Silicon}\label{md__ase_sprite_aseprite__i_n_s_t_a_l_l_autotoc_md79}
If you running mac\+OS on an ARM64/\+AArch64/\+Apple Silicon Mac (e.\+g. M1), you can compile a native ARM64 version of Aseprite following the same steps as above but when we call {\ttfamily cmake}, we have some differences\+: \begin{DoxyVerb}cmake \

  -DCMAKE_OSX_ARCHITECTURES=arm64 \
  -DCMAKE_OSX_DEPLOYMENT_TARGET=11.0 \
  -DCMAKE_OSX_SYSROOT=/Applications/Xcode.app/Contents/Developer/Platforms/MacOSX.platform/Developer/SDKs/MacOSX.sdk \
  -DLAF_BACKEND=skia \
  -DSKIA_DIR=$HOME/deps/skia \
  -DSKIA_LIBRARY_DIR=$HOME/deps/skia/out/Release-arm64 \
  -DSKIA_LIBRARY=$HOME/deps/skia/out/Release-arm64/libskia.a \
  -DPNG_ARM_NEON:STRING=on \

  ..
\end{DoxyVerb}

\href{https://github.com/aseprite/aseprite/issues/589}{\texttt{ https\+://github.\+com/aseprite/aseprite/issues/589}}\hypertarget{md__ase_sprite_aseprite__i_n_s_t_a_l_l_autotoc_md81}{}\doxysubsection{Linux details}\label{md__ase_sprite_aseprite__i_n_s_t_a_l_l_autotoc_md81}
You need to use clang and libc++ to compile Aseprite\+: \begin{DoxyVerb}cd aseprite

cd build
export CC=clang
export CXX=clang++

  -DCMAKE_BUILD_TYPE=RelWithDebInfo \
  -DCMAKE_CXX_FLAGS:STRING=-stdlib=libc++ \
  -DCMAKE_EXE_LINKER_FLAGS:STRING=-stdlib=libc++ \
  -DLAF_BACKEND=skia \
  -DSKIA_DIR=$HOME/deps/skia \
  -DSKIA_LIBRARY_DIR=$HOME/deps/skia/out/Release-x64 \
  -DSKIA_LIBRARY=$HOME/deps/skia/out/Release-x64/libskia.a \
  -G Ninja \
  ..
ninja aseprite
\end{DoxyVerb}
 In this case, {\ttfamily \$HOME/deps/skia} is the directory where Skia was compiled or uncompressed.\hypertarget{md__ase_sprite_aseprite__i_n_s_t_a_l_l_autotoc_md82}{}\doxysubsubsection{GCC compiler}\label{md__ase_sprite_aseprite__i_n_s_t_a_l_l_autotoc_md82}
In case that you are using the pre-\/compiled Skia version, you must use the clang compiler and libc++ to compile Aseprite. Only if you compile Skia with GCC, you will be able to compile Aseprite with GCC, and this is not recommended as you will have a performance penalty doing so.\hypertarget{md__ase_sprite_aseprite__i_n_s_t_a_l_l_autotoc_md83}{}\doxysection{Using shared third party libraries}\label{md__ase_sprite_aseprite__i_n_s_t_a_l_l_autotoc_md83}
If you don\textquotesingle{}t want to use the embedded code of third party libraries (i.\+e. to use your installed versions), you can disable static linking configuring each {\ttfamily USE\+\_\+\+SHARED\+\_\+} option.

After running {\ttfamily cmake -\/G}, you can edit {\ttfamily build/\+CMake\+Cache.\+txt} file, and enable the {\ttfamily USE\+\_\+\+SHARED\+\_\+} flag (set its value to {\ttfamily ON}) of the library that you want to be linked dynamically.
