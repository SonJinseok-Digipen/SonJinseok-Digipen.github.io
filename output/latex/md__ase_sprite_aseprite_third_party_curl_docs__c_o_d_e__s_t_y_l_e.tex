Source code that has a common style is easier to read than code that uses different styles in different places. It helps making the code feel like one single code base. Easy-\/to-\/read is a very important property of code and helps making it easier to review when new things are added and it helps debugging code when developers are trying to figure out why things go wrong. \mbox{\hyperlink{class_a}{A}} unified style is more important than individual contributors having their own personal tastes satisfied.

Our C code has a few style rules. Most of them are verified and upheld by the {\ttfamily lib/checksrc.\+pl} script. Invoked with {\ttfamily make checksrc} or even by default by the build system when built after {\ttfamily ./configure -\/-\/enable-\/debug} has been used.

It is normally not a problem for anyone to follow the guidelines, as you just need to copy the style already used in the source code and there are no particularly unusual rules in our set of rules.

We also work hard on writing code that are warning-\/free on all the major platforms and in general on as many platforms as possible. Code that obviously will cause warnings will not be accepted as-\/is.\hypertarget{md__ase_sprite_aseprite_third_party_curl_docs__c_o_d_e__s_t_y_l_e_autotoc_md772}{}\doxysection{Naming}\label{md__ase_sprite_aseprite_third_party_curl_docs__c_o_d_e__s_t_y_l_e_autotoc_md772}
Try using a non-\/confusing naming scheme for your new functions and variable names. It doesn\textquotesingle{}t necessarily have to mean that you should use the same as in other places of the code, just that the names should be logical, understandable and be named according to what they\textquotesingle{}re used for. File-\/local functions should be made static. We like lower case names.

See the \href{https://curl.se/dev/internals.html\#symbols}{\texttt{ INTERNALS}} document on how we name non-\/exported library-\/global symbols.\hypertarget{md__ase_sprite_aseprite_third_party_curl_docs__c_o_d_e__s_t_y_l_e_autotoc_md773}{}\doxysection{Indenting}\label{md__ase_sprite_aseprite_third_party_curl_docs__c_o_d_e__s_t_y_l_e_autotoc_md773}
We use only spaces for indentation, never TABs. We use two spaces for each new open brace.


\begin{DoxyCode}{0}
\DoxyCodeLine{\textcolor{keywordflow}{if}(something\_is\_true)\ \{}
\DoxyCodeLine{\ \ \textcolor{keywordflow}{while}(second\_statement\ ==\ fine)\ \{}
\DoxyCodeLine{\ \ \ \ moo();}
\DoxyCodeLine{\ \ \}}
\DoxyCodeLine{\}}

\end{DoxyCode}
\hypertarget{md__ase_sprite_aseprite_third_party_curl_docs__c_o_d_e__s_t_y_l_e_autotoc_md774}{}\doxysection{Comments}\label{md__ase_sprite_aseprite_third_party_curl_docs__c_o_d_e__s_t_y_l_e_autotoc_md774}
Since we write C89 code, $\ast$$\ast$//$\ast$$\ast$ comments are not allowed. They weren\textquotesingle{}t introduced in the C standard until C99. We use only \+\_\+\+\_\+/$\ast$ comments $\ast$/\+\_\+\+\_\+.


\begin{DoxyCode}{0}
\DoxyCodeLine{\textcolor{comment}{/*\ this\ is\ a\ comment\ */}}

\end{DoxyCode}
\hypertarget{md__ase_sprite_aseprite_third_party_curl_docs__c_o_d_e__s_t_y_l_e_autotoc_md775}{}\doxysection{Long lines}\label{md__ase_sprite_aseprite_third_party_curl_docs__c_o_d_e__s_t_y_l_e_autotoc_md775}
Source code in curl may never be wider than 79 columns and there are two reasons for maintaining this even in the modern era of very large and high resolution screens\+:


\begin{DoxyEnumerate}
\item Narrower columns are easier to read than very wide ones. There\textquotesingle{}s a reason newspapers have used columns for decades or centuries.
\item Narrower columns allow developers to easier show multiple pieces of code next to each other in different windows. I often have two or three source code windows next to each other on the same screen -\/ as well as multiple terminal and debugging windows.
\end{DoxyEnumerate}\hypertarget{md__ase_sprite_aseprite_third_party_curl_docs__c_o_d_e__s_t_y_l_e_autotoc_md776}{}\doxysection{Braces}\label{md__ase_sprite_aseprite_third_party_curl_docs__c_o_d_e__s_t_y_l_e_autotoc_md776}
In if/while/do/for expressions, we write the open brace on the same line as the keyword and we then set the closing brace on the same indentation level as the initial keyword. Like this\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{\textcolor{keywordflow}{if}(age\ <\ 40)\ \{}
\DoxyCodeLine{\ \ \textcolor{comment}{/*\ clearly\ a\ youngster\ */}}
\DoxyCodeLine{\}}

\end{DoxyCode}


You may omit the braces if they would contain only a one-\/line statement\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{\textcolor{keywordflow}{if}(!x)}
\DoxyCodeLine{\ \ \textcolor{keywordflow}{continue};}

\end{DoxyCode}


For functions the opening brace should be on a separate line\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{\textcolor{keywordtype}{int}\ main(\textcolor{keywordtype}{int}\ argc,\ \textcolor{keywordtype}{char}\ **argv)}
\DoxyCodeLine{\{}
\DoxyCodeLine{\ \ \textcolor{keywordflow}{return}\ 1;}
\DoxyCodeLine{\}}

\end{DoxyCode}
\hypertarget{md__ase_sprite_aseprite_third_party_curl_docs__c_o_d_e__s_t_y_l_e_autotoc_md777}{}\doxysection{\textquotesingle{}else\textquotesingle{} on the following line}\label{md__ase_sprite_aseprite_third_party_curl_docs__c_o_d_e__s_t_y_l_e_autotoc_md777}
When adding an {\bfseries{else}} clause to a conditional expression using braces, we add it on a new line after the closing brace. Like this\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{\textcolor{keywordflow}{if}(age\ <\ 40)\ \{}
\DoxyCodeLine{\ \ \textcolor{comment}{/*\ clearly\ a\ youngster\ */}}
\DoxyCodeLine{\}}
\DoxyCodeLine{\textcolor{keywordflow}{else}\ \{}
\DoxyCodeLine{\ \ \textcolor{comment}{/*\ probably\ grumpy\ */}}
\DoxyCodeLine{\}}

\end{DoxyCode}
\hypertarget{md__ase_sprite_aseprite_third_party_curl_docs__c_o_d_e__s_t_y_l_e_autotoc_md778}{}\doxysection{No space before parentheses}\label{md__ase_sprite_aseprite_third_party_curl_docs__c_o_d_e__s_t_y_l_e_autotoc_md778}
When writing expressions using if/while/do/for, there shall be no space between the keyword and the open parenthesis. Like this\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{\textcolor{keywordflow}{while}(1)\ \{}
\DoxyCodeLine{\ \ \textcolor{comment}{/*\ loop\ forever\ */}}
\DoxyCodeLine{\}}

\end{DoxyCode}
\hypertarget{md__ase_sprite_aseprite_third_party_curl_docs__c_o_d_e__s_t_y_l_e_autotoc_md779}{}\doxysection{Use boolean conditions}\label{md__ase_sprite_aseprite_third_party_curl_docs__c_o_d_e__s_t_y_l_e_autotoc_md779}
Rather than test a conditional value such as a bool against TRUE or FALSE, a pointer against NULL or != NULL and an int against zero or not zero in if/while conditions we prefer\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{result\ =\ do\_something();}
\DoxyCodeLine{\textcolor{keywordflow}{if}(!result)\ \{}
\DoxyCodeLine{\ \ \textcolor{comment}{/*\ something\ went\ wrong\ */}}
\DoxyCodeLine{\ \ \textcolor{keywordflow}{return}\ result;}
\DoxyCodeLine{\}}

\end{DoxyCode}
\hypertarget{md__ase_sprite_aseprite_third_party_curl_docs__c_o_d_e__s_t_y_l_e_autotoc_md780}{}\doxysection{No assignments in conditions}\label{md__ase_sprite_aseprite_third_party_curl_docs__c_o_d_e__s_t_y_l_e_autotoc_md780}
To increase readability and reduce complexity of conditionals, we avoid assigning variables within if/while conditions. We frown upon this style\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{\textcolor{keywordflow}{if}((ptr\ =\ malloc(100))\ ==\ NULL)}
\DoxyCodeLine{\ \ \textcolor{keywordflow}{return}\ NULL;}

\end{DoxyCode}


and instead we encourage the above version to be spelled out more clearly\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{ptr\ =\ malloc(100);}
\DoxyCodeLine{\textcolor{keywordflow}{if}(!ptr)}
\DoxyCodeLine{\ \ \textcolor{keywordflow}{return}\ NULL;}

\end{DoxyCode}
\hypertarget{md__ase_sprite_aseprite_third_party_curl_docs__c_o_d_e__s_t_y_l_e_autotoc_md781}{}\doxysection{New block on a new line}\label{md__ase_sprite_aseprite_third_party_curl_docs__c_o_d_e__s_t_y_l_e_autotoc_md781}
We never write multiple statements on the same source line, even for very short if() conditions.


\begin{DoxyCode}{0}
\DoxyCodeLine{\textcolor{keywordflow}{if}(\mbox{\hyperlink{structa}{a}})}
\DoxyCodeLine{\ \ \textcolor{keywordflow}{return}\ TRUE;}
\DoxyCodeLine{\textcolor{keywordflow}{else}\ \textcolor{keywordflow}{if}(b)}
\DoxyCodeLine{\ \ \textcolor{keywordflow}{return}\ FALSE;}

\end{DoxyCode}


and NEVER\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{\textcolor{keywordflow}{if}(\mbox{\hyperlink{structa}{a}})\ \textcolor{keywordflow}{return}\ TRUE;}
\DoxyCodeLine{\textcolor{keywordflow}{else}\ \textcolor{keywordflow}{if}(b)\ \textcolor{keywordflow}{return}\ FALSE;}

\end{DoxyCode}
\hypertarget{md__ase_sprite_aseprite_third_party_curl_docs__c_o_d_e__s_t_y_l_e_autotoc_md782}{}\doxysection{Space around operators}\label{md__ase_sprite_aseprite_third_party_curl_docs__c_o_d_e__s_t_y_l_e_autotoc_md782}
Please use spaces on both sides of operators in C expressions. Postfix $\ast$$\ast$(), \mbox{[}\mbox{]}, -\/\texorpdfstring{$>$}{>}, ., ++, --$\ast$$\ast$ and Unary {\bfseries{+, -\/, !, $\sim$, \&}} operators excluded they should have no space.

Examples\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{bla\ =\ func();}
\DoxyCodeLine{who\ =\ name[0];}
\DoxyCodeLine{age\ +=\ 1;}
\DoxyCodeLine{\textcolor{keyword}{true}\ =\ !\textcolor{keyword}{false};}
\DoxyCodeLine{size\ +=\ -\/2\ +\ 3\ *\ (\mbox{\hyperlink{structa}{a}}\ +\ b);}
\DoxyCodeLine{ptr-\/>member\ =\ \mbox{\hyperlink{structa}{a}}++;}
\DoxyCodeLine{\textcolor{keyword}{struct}.field\ =\ b-\/-\/;}
\DoxyCodeLine{ptr\ =\ \&address;}
\DoxyCodeLine{\mbox{\hyperlink{structcontents}{contents}}\ =\ *pointer;}
\DoxyCodeLine{complement\ =\ \string~bits;}
\DoxyCodeLine{empty\ =\ (!*string)\ ?\ TRUE\ :\ FALSE;}

\end{DoxyCode}
\hypertarget{md__ase_sprite_aseprite_third_party_curl_docs__c_o_d_e__s_t_y_l_e_autotoc_md783}{}\doxysection{No parentheses for return values}\label{md__ase_sprite_aseprite_third_party_curl_docs__c_o_d_e__s_t_y_l_e_autotoc_md783}
We use the \textquotesingle{}return\textquotesingle{} statement without extra parentheses around the value\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{\textcolor{keywordtype}{int}\ works(\textcolor{keywordtype}{void})}
\DoxyCodeLine{\{}
\DoxyCodeLine{\ \ \textcolor{keywordflow}{return}\ TRUE;}

\end{DoxyCode}
\hypertarget{md__ase_sprite_aseprite_third_party_curl_docs__c_o_d_e__s_t_y_l_e_autotoc_md784}{}\doxysection{Parentheses for sizeof arguments}\label{md__ase_sprite_aseprite_third_party_curl_docs__c_o_d_e__s_t_y_l_e_autotoc_md784}
When using the sizeof operator in code, we prefer it to be written with parentheses around its argument\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{\textcolor{keywordtype}{int}\ size\ =\ \textcolor{keyword}{sizeof}(int);}

\end{DoxyCode}
\hypertarget{md__ase_sprite_aseprite_third_party_curl_docs__c_o_d_e__s_t_y_l_e_autotoc_md785}{}\doxysection{Column alignment}\label{md__ase_sprite_aseprite_third_party_curl_docs__c_o_d_e__s_t_y_l_e_autotoc_md785}
Some statements cannot be completed on a single line because the line would be too long, the statement too hard to read, or due to other style guidelines above. In such a case the statement will span multiple lines.

If a continuation line is part of an expression or sub-\/expression then you should align on the appropriate column so that it\textquotesingle{}s easy to tell what part of the statement it is. Operators should not start continuation lines. In other cases follow the 2-\/space indent guideline. Here are some examples from libcurl\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{\textcolor{keywordflow}{if}(Curl\_pipeline\_wanted(handle-\/>multi,\ CURLPIPE\_HTTP1)\ \&\&}

\DoxyCodeLine{\ \ \textcolor{keywordflow}{return}\ TRUE;}

\end{DoxyCode}

\begin{DoxyCode}{0}

\end{DoxyCode}

\DoxyCodeLine{\}}

\end{DoxyCode}

\begin{DoxyCode}{0}
\DoxyCodeLine{DEBUGF(infof(\mbox{\hyperlink{structdata}{data}},\ \textcolor{stringliteral}{"{}Curl\_pp\_readresp\_\ \%d\ bytes\ of\ trailing\ "{}}}

\hypertarget{md__ase_sprite_aseprite_third_party_curl_docs__c_o_d_e__s_t_y_l_e_autotoc_md786}{}\doxysection{Platform dependent code}\label{md__ase_sprite_aseprite_third_party_curl_docs__c_o_d_e__s_t_y_l_e_autotoc_md786}
Use $\ast$$\ast$\#ifdef HAVE\+\_\+\+FEATURE$\ast$$\ast$ to do conditional code. We avoid checking for particular operating systems or hardware in the \#ifdef lines. The HAVE\+\_\+\+FEATURE shall be generated by the configure script for unix-\/like systems and they are hard-\/coded in the {\ttfamily config-\/\mbox{[}system\mbox{]}.h} files for the others.

We also encourage use of macros/functions that possibly are empty or defined to constants when libcurl is built without that feature, to make the code seamless. Like this example where the {\bfseries{magic()}} function works differently depending on a build-\/time conditional\+:


\begin{DoxyCode}{0}

\DoxyCodeLine{\}}

\DoxyCodeLine{\textcolor{preprocessor}{\#define\ magic(x)\ 1}}
\DoxyCodeLine{\textcolor{preprocessor}{\#endif}}

\DoxyCodeLine{\textcolor{keywordtype}{int}\ \mbox{\hyperlink{structcontent}{content}}\ =\ magic(3);}

\end{DoxyCode}
